# Multi-Sample Implementation Patterns

How to decide implementation form for Self-Consistency / multi-sample / ensemble LLM patterns.

## When Multi-Sample Documentation Required

If the workflow uses any of:

- **Self-Consistency**: N independent LLM calls with variance source, then integration
- **Ensemble LLM**: N different models on same input
- **Multi-Judge Eval**: N evaluators on same output
- **N-Best Decoding + Reranking**: top-N candidates + ranking model
- **Bayesian Model Averaging**: weighted average of N model predictions

then workflow.md MUST add the Multi-Sample Implementation Pattern block to the Inherited DNA section.

## Required Block Structure

```markdown
### Multi-Sample Implementation Pattern

- **Pattern**: <Sequential N+1 calls | Agent Team N parallel + Team Lead integration | Hybrid>
- **N**: <integer>
- **Variance source**: <temperature | prompt_variation | seed | ensemble_model_diversity | other>
- **Integration mechanism**: <T=X.X integration call | majority vote | weighted average | LLM judge | none>
- **Trade-off rationale**: <quality vs throughput vs rate_limit vs cost — explicit choice>
- **Cost impact**: <integrated with cost_budget per affected stage>
- **Failure mode**: <strategy when sample variance exceeds threshold>
```

## Pattern Selection Decision

### Pattern 1: `Sequential N+1 calls`

N sample calls executed sequentially, then 1 integration call (or no integration if voting).

**Trade-offs**:
- Latency: N+1 × per-call latency (high)
- Throughput: low (1 execution at a time)
- Rate limit pressure: distributed (no spike)
- Implementation simplicity: high
- SOT design: simple (single Sub-agent)

**Use when**:
- Rate limit is the binding constraint
- Latency budget is loose
- Implementation simplicity is valued
- Per-call quality > parallelization throughput

**Implementation primitive**:
- Sub-agent with N+1 sequential `call_llm()` invocations

### Pattern 2: `Agent Team N parallel + Team Lead integration`

N samples generated by parallel teammates; Team Lead integrates results.

**Trade-offs**:
- Latency: max(per-call) + integration (low)
- Throughput: high if rate limit allows
- Rate limit pressure: spike (N concurrent)
- Implementation complexity: higher (SOT design required)
- SOT design: Absolute Criterion 2 (Team Lead single write)

**Use when**:
- Latency is the binding constraint
- Rate limit tier supports concurrent calls (subscription, dedicated)
- Multi-perspective parallel work improves quality
- Throughput matters

**Implementation primitive**:
- Agent Team with N parallel teammates + 1 Team Lead integrator
- SOT schema: `active_team` field per workflow-generator template

### Pattern 3: `Hybrid` (e.g., Sequential N=2 batch parallel)

Mix of sequential and parallel (e.g., 2 batches of N/2 parallel calls in series).

**Trade-offs**:
- Latency: ⌈N/B⌉ × per-call + integration (B = batch size)
- Throughput: medium
- Rate limit pressure: bounded by batch size
- Implementation complexity: high

**Use when**:
- Rate limit is partial constraint
- Latency budget is moderate
- Quality benefits from N samples but rate limit forbids full parallelization

**Implementation primitive**:
- Agent Team with bounded concurrency + Team Lead integration

## Variance Source Selection

### Temperature

LLM temperature parameter sets randomness in token sampling.

**Use when**:
- LLM API exposes temperature
- Variance from token-level sampling is desired (typical)

**Caution**: some subscription channels do not expose temperature; verify availability before commitment.

### Prompt Variation

Different system prompts (paraphrased) produce different outputs.

**Use when**:
- Temperature unavailable (fallback)
- Want to probe prompt-engineering robustness, not stochasticity
- Larger semantic variance desired than temperature provides

**Note**: Variance source semantically differs from temperature variance — eval threshold must be re-derived.

### Seed

Fixed seed with controlled variation (e.g., seed=1, 2, 3).

**Use when**:
- Reproducibility of variance is required
- Debugging or testing the variance pattern itself

### Ensemble Model Diversity

N different model architectures or sizes.

**Use when**:
- Cross-architecture variance reduces shared bias
- Cost of model diversity is acceptable
- Quality is paramount

## Integration Mechanism Selection

### `T=X.X integration call`

Lower-temperature LLM call merging N generations into single coherent output.

**Use when**:
- Output requires coherent narrative (not vote-able items)
- Quality of merged output > simple voting
- One additional call cost acceptable

**Trade-off**: integration call introduces additional LLM cost + latency.

### `Majority vote`

Discrete output: pick most common value across N samples.

**Use when**:
- Output is categorical / discrete (label, class, decision)
- Coherent narrative not required
- Simplicity preferred

### `Weighted average`

Continuous output: weighted mean across N samples.

**Use when**:
- Output is numeric (score, probability, confidence)
- Weights derived from sample quality / consistency

### `LLM judge`

Separate LLM scores N samples, picks best (or weighted combination).

**Use when**:
- Quality of each sample varies non-trivially
- Domain expert LLM can score reliably
- N is small enough that judge cost is acceptable

**Trade-off**: judge introduces additional LLM cost + potential bias.

### `None` (no integration)

N samples are reported separately; downstream consumes all.

**Use when**:
- Each sample has independent value (e.g., diversity desired)
- Downstream phase will integrate (deferred integration)

## Trade-Off Rationale Discipline

When documenting trade-off, name explicit dimensions:

```markdown
**Trade-off rationale**: Pattern selection rationale.
- Quality dimension: <X> (e.g., per-call deep-context vs multi-perspective diversity)
- Throughput dimension: <Y> (e.g., latency budget X ms; pattern A latency Y ms; pattern B latency Z ms)
- Rate limit dimension: <Z> (e.g., subscription tier X; pattern A consumes 1 RPM; pattern B consumes N RPM)
- Cost dimension: <W> (e.g., pattern A: N+1 calls; pattern B: N+2 calls including Team Lead integration)
- Final choice: <Pattern>
- Trumping criterion: <which dimension was decisive — typically Quality per Absolute Criterion 1>
```

## Failure Mode (Variance Threshold Breach)

When sample variance exceeds expected threshold, document escalation:

| Failure Mode | Strategy |
|---|---|
| `regenerate_full_set` | Discard N samples, regenerate (1 retry) |
| `extend_sample_set` | Add M additional samples (N → N+M); recompute variance |
| `flag_for_human_review` | Surface to user; mark output as low-confidence |
| `fallback_to_single_call` | Abandon multi-sample; single deterministic-prompt call |
| `escalate_to_judge` | Use LLM judge to pick best of N (override integration) |

## Examples Across Domains

### Implementation Pipeline LLM Module (any workflow)

```markdown
### Multi-Sample Implementation Pattern

- **Pattern**: Sequential N+1 calls
- **N**: 3
- **Variance source**: temperature (T=0.9 for generation, T=0.3 for integration)
- **Integration mechanism**: T=0.3 integration call merging N=3 generations into single coherent output
- **Trade-off rationale**:
  - Quality: deep coherent narrative > N=3 fragmented outputs
  - Throughput: latency budget 30s p95; sequential 4 × 5s = 20s; parallel 5s + 5s = 10s but rate limit 1 RPM forbids parallel
  - Rate limit: subscription channel 1 RPM hard limit (R-003 awareness)
  - Cost: N+1 = 4 calls per section; matches Cost-Latency Policy budget
  - Final: Sequential N+1 — quality + rate-limit-feasible
- **Cost impact**: 4 calls per stage × 33 sections = 132 calls per execution (median)
- **Failure mode**: deviation > 5% triggers regenerate_full_set (1 retry); persistent breach triggers flag_for_human_review per <upstream spec>
```

### Multi-Judge LLM Eval

```markdown
### Multi-Sample Implementation Pattern

- **Pattern**: Agent Team N parallel + Team Lead integration
- **N**: 3 (3 judge LLMs)
- **Variance source**: ensemble_model_diversity (3 different model checkpoints)
- **Integration mechanism**: weighted average + disagreement flag
- **Trade-off rationale**:
  - Quality: cross-model bias reduction
  - Throughput: parallel feasible per API tier
  - Cost: 3 judge calls + 1 aggregation = 4 calls per evaluated unit
- **Cost impact**: 4 calls per evaluation × M evaluations
- **Failure mode**: judge disagreement > threshold flags for human reviewer (escalate_to_judge fallback to senior model)
```

### N-Best Decoding + Reranking

```markdown
### Multi-Sample Implementation Pattern

- **Pattern**: Sequential (single call returning N candidates) + 1 reranker call
- **N**: 5 (n_best from beam search)
- **Variance source**: beam_search_diversity (built into LLM)
- **Integration mechanism**: LLM judge reranking
- **Trade-off rationale**:
  - Quality: top-1 from N=5 reranked > top-1 single sample
  - Cost: N candidates from single call (no extra cost) + 1 reranker call
- **Cost impact**: 1 generation call + 1 reranker call = 2 calls per query
- **Failure mode**: reranker score < confidence threshold → flag_for_human_review
```

### Ensemble ML Inference

```markdown
### Multi-Sample Implementation Pattern

- **Pattern**: Agent Team N parallel + Team Lead integration (deterministic aggregator)
- **N**: 5 (5 model checkpoints)
- **Variance source**: ensemble_model_diversity (different training seeds)
- **Integration mechanism**: weighted average (per-model uncertainty-weighted)
- **Trade-off rationale**:
  - Quality: ensemble reduces model variance
  - Cost: 5 model inferences (each cheap)
- **Cost impact**: 5 inferences per input
- **Failure mode**: ensemble disagreement > threshold (entropy > X) → flag_for_human_review
```

### Bayesian Model Averaging

```markdown
### Multi-Sample Implementation Pattern

- **Pattern**: Sequential (per posterior sample) — typically N=100+
- **N**: 100
- **Variance source**: posterior_distribution_sampling
- **Integration mechanism**: weighted average (Bayes posterior weights)
- **Trade-off rationale**: Bayesian uncertainty quantification > point estimate
- **Cost impact**: 100 model evaluations per input (cheap models only)
- **Failure mode**: posterior variance > threshold → extend_sample_set
```

## Cross-Document Validation

After multi-sample pattern documented, verify:

```python
def validate_multi_sample(workflow):
    if not workflow.uses_multi_sample:
        return  # not applicable

    # Block must exist in Inherited DNA
    assert "Multi-Sample Implementation Pattern" in workflow.inherited_dna

    # Cost impact must align with cost_budget per affected stage
    for stage in workflow.stages:
        if stage.uses_multi_sample:
            assert stage.cost_budget.llm_calls >= workflow.multi_sample.N + (1 if integration else 0)

    # Pattern must align with rate-limit feasibility
    if workflow.multi_sample.pattern == "Agent Team N parallel + Team Lead integration":
        assert workflow.cost_budget.rate_limit_tier in ["subscription", "dedicated", "api_tier_X+"]
        assert "active_team" in workflow.sot_schema  # SOT design required per Absolute Criterion 2

    # Failure mode must be specified
    assert workflow.multi_sample.failure_mode is not None
```

## Anti-Patterns

| Anti-Pattern | Why Forbidden | Fix |
|---|---|---|
| Multi-sample used without documentation block | Pattern undocumented; downstream cannot reproduce | Add Multi-Sample Implementation Pattern block |
| Pattern selection without trade-off rationale | Choice opaque; cannot revisit | Document quality/throughput/rate-limit/cost dimensions |
| Agent Team parallel without rate-limit tier verification | Production rate-limit failure | Verify rate-limit tier capacity vs N concurrent |
| Variance source = temperature without API capability check | Temperature unavailable runtime failure | Probe variance source feasibility before commit |
| No failure mode for variance threshold breach | Production silent quality breach | Specify failure mode strategy |
| Integration mechanism = "none" when downstream expects integrated output | Output incompatibility | Match integration mechanism to downstream expectation |
| N too small for statistical bound | Variance unreliable | Increase N or document quality limitation |
| Sequential pattern when latency budget tight + rate limit allows parallel | Latency budget violation | Switch to Agent Team parallel |
